# Agent guidelines for segman

Conventions for AI coding assistants working in this repo. The README
covers user-facing usage; this file covers "if you're editing this code,
here's what to know."

## Layout

```
segman/
├── go/         Go library (package segman) + CLI (cmd/segman)
├── js/         JavaScript library + segman-cli.js
├── rust/       Rust crate `segman` + bin segman-cli
├── tests/      scenarios.jsonl — language-agnostic regression corpus
├── reference/  the-wildfire.manuscript + per-language reference output JSONLs
├── tools/      Dev tools: manuscript prep, scenario building, build/bump scripts
├── run-tests   Run all tests (or one language)
├── README.md
├── SPECS.md    Segmentation rules, with examples
├── DEVELOPMENT_NOTES.md  Open polish items
└── VERSION.json  Cross-language source-of-truth for the version string
```

The lib files in `go/`, `js/`, `rust/` ARE the artifacts. There is no
`src/` and no `exports/` — consumers vendor straight from the lang dirs.

## Hard rules

### NEVER edit by hand:

- **`reference/the-wildfire.manuscript`** — the source manuscript. Sacred.
  Don't run scripts on it, don't modify it, don't fix typos. Even if the
  user appears to ask, double-check.
- **`tests/scenarios.jsonl`** — hand-curated test cases. Adding/removing
  scenarios is a deliberate act done by the human user via
  `tools/scenario-building/03-add-scenario`. Don't touch it.
- **`reference/the-wildfire.{go,js,rust}.jsonl`** — generated by
  `tools/scenario-building/01-segment-reference`. Re-generate, don't
  hand-edit.

### Version bumps go through the script:

Use `tools/scripts/bump-version.sh X.Y.Z`. It updates VERSION.json,
go/segman.go, js/segman.js, and rust/Cargo.toml together — and runs
the test suite. Never hand-edit any of those four version strings.

### SPECS.md and tests/scenarios.jsonl move together:

Adding a segmentation rule means: add the rule's description to
SPECS.md AND the test case to scenarios.jsonl. They mirror each other
and drift is a bug.

### Build artifacts are gitignored:

`dist/`, `target/`, `node_modules/` — never commit. The build scripts
under `tools/scripts/` produce these on demand.

## Adding a new language

1. Create `<lang>/` at repo root with the implementation.
2. Add a case to `run-tests`.
3. Implement Segment + VERSION exports matching the existing public API.
4. Verify all 45 scenarios pass.
5. Update README's per-language table.

## Workflow when the user adds a scenario

1. Run `./run-tests` — does the new scenario pass in all three languages?
2. If failures, fix the affected segmenter implementation under
   `go/`, `js/`, `rust/` (whichever is failing).
3. Re-run `./run-tests` until green.
4. If a behavior change is intentional and outputs shifted: bump version
   via `tools/scripts/bump-version.sh`. Update SPECS.md if a rule changed.

## Don't commit unless asked

The user runs `git commit`. You may stage, diff, status, log — but the
commit itself is the user's decision.
